Section I

Introduction

1. Contexte

Le secteur pharmaceutique est l'un des domaines les plus exigeants en matière de rigueur et de traçabilité. La gestion des stocks de médicaments, le suivi des dates d'expiration, la facturation des ventes et la gestion des crédits clients constituent des activités critiques qui, lorsqu'elles sont réalisées manuellement, génèrent des erreurs humaines coûteuses et représentent un risque pour la santé publique.

Dans ce contexte, la transformation numérique des pharmacies s'impose comme une nécessité stratégique. Le développement d'une application web sur mesure — telle que 7X Pharma Nexus — répond directement à ce besoin en proposant une solution centralisée, structurée et sécurisée pour la gestion quotidienne d'une pharmacie.

2. Problématique

Les pharmacies traditionnelles font face à plusieurs défis structurels :

  • Difficulté à suivre les dates d'expiration des médicaments, exposant la pharmacie à des risques financiers et sanitaires.
  • Gestion inefficace et non-traçable des crédits accordés aux clients réguliers.
  • Lenteur lors du processus de vente et de facturation, impactant la satisfaction client.
  • Manque de visibilité sur les statistiques journalières (recettes, nombre de transactions, alertes de stock).

La problématique centrale de ce projet est donc :

Comment concevoir et développer une application web pharmaceutique complète, sécurisée et performante, permettant de numériser la gestion des ventes, des stocks et des crédits d'une pharmacie, tout en garantissant la traçabilité des opérations ?

3. Objectifs du Projet

L'objectif principal de ce projet est de concevoir et réaliser une plateforme web pharmaceutique répondant aux besoins identifiés. Les objectifs spécifiques sont :

  • Créer une interface de vente (POS) intuitive et rapide, utilisant un système de panier basé sur les sessions PHP.
  • Automatiser les alertes pour les médicaments dont le stock est inférieur à 10 unités.
  • Gérer les paiements à crédit et conserver un historique détaillé des transactions clients.
  • Implémenter une logique FIFO (First In, First Out) pour la déduction automatique du stock à partir des lots les plus anciens.
  • Offrir un tableau de bord analytique affichant les indicateurs clés en temps réel.

Section IV

Conception

Après l'analyse des besoins, la phase de conception permet de définir l'architecture du système, les interactions entre acteurs et composants, ainsi que la structure de la base de données avant la réalisation.

1. Architecture Générale du Système

7X Pharma Nexus est fondé sur une architecture MVC (Modèle-Vue-Contrôleur) personnalisée, déployée selon le schéma client-serveur classique. Le flux de traitement est strictement unidirectionnel :

  1. Le client envoie une requête HTTP via le navigateur web.
  2. Le serveur Apache redirige la requête (via .htaccess) vers index.php.
  3. Le Router instancie le bon Contrôleur selon l'URL analysée.
  4. Le Contrôleur demande les données au Modèle, qui interroge la base de données.
  5. Les données sont retournées au Contrôleur, qui les passe à la Vue.
  6. La Vue génère le HTML final et le renvoie au navigateur.

Description des composants :

  • Modèles (app/models/) : Héritent de la classe de base Model.php. Ils encapsulent la logique des données et les requêtes SQL (ex: Medicine.php, Sale.php).
  • Vues (views/pages/) : Fichiers HTML/PHP qui génèrent l'interface utilisateur à partir des données fournies par les contrôleurs (ex: dashboard.php, pos.php).
  • Contrôleurs (app/controllers/) : Héritent de Controller.php. Ils traitent les requêtes HTTP (GET/POST) et appliquent la logique métier.
  • Base de données : Stocke les médicaments, lots, ventes, articles, clients, paiements et mouvements de stock.

2. Système de Routage

Le fichier Router.php, situé dans app/core/, constitue le cœur du système de navigation. Il capture l'URL (ex: /inventory/edit/5), la nettoie et la découpe en tableau. Le premier élément définit le contrôleur (ex: InventoryController), le deuxième la méthode (ex: edit), et le reste les paramètres (ex: 5). Si aucun contrôleur n'est spécifié, le système charge POSController par défaut.

4. Conception de la Base de Données

La base de données pharma_core est conçue avec MySQL/MariaDB et connectée via PDO dans Database.php. Elle repose sur un modèle relationnel complet garantissant l'intégrité des données à travers les clés étrangères.

Tables Principales :

Table users — Gestion des accès utilisateurs.

  • id : Identifiant unique.
  • username : Nom d'utilisateur.
  • password : Mot de passe haché (BCRYPT).
  • role : Rôle de l'utilisateur (pharmacien ou administrateur).

Cette table contrôle les droits d'accès à l'ensemble de l'application.

Table medicines — Catalogue des médicaments.

  • id, barcode, name, dci, category, price, cost_price.
  • deleted_at : Permet le soft delete sans perte définitive des données.

Cette table stocke les informations de base de chaque médicament, indépendamment de son stock physique.

Table batches — Gestion physique du stock par lots.

  • medicine_id : Lien vers le médicament parent.
  • batch_number, expiry_date, current_quantity.

Chaque lot représente une livraison distincte avec sa propre date d'expiration. La logique FIFO utilise cette table pour vider les lots les plus anciens en premier.

Table sales — En-têtes de factures.

  • user_id, client_id, total_amount, payment_method.

Table sale_items — Lignes de chaque facture.

  • batch_id, quantity, unit_price.
  • snapshot_data (JSON) : Fige le nom et le prix du médicament au moment exact de la vente pour l'archivage historique.

Table clients — Clients réguliers avec crédit.

  • name, phone, credit_balance.

Table client_payments — Historique des règlements de dette.

  • client_id, amount, payment_method, user_id, payment_date.

Table inventory_movements — Journal de tous les mouvements de stock (entrée, vente, retour fournisseur, produits périmés, ajustements).

ضع هنا مخطط قاعدة البيانات الكامل (MCD / Schéma Relationnel) يوضح جميع الجداول والعلاقات بينها

Relations entre les Tables :

  • Un Médicament (medicines) a plusieurs Lots (batches).
  • Une Vente (sales) a plusieurs Articles (sale_items). Chaque article pointe vers un lot spécifique via batch_id.
  • Une Vente peut être liée à un Client si elle est réglée à crédit.
  • Un Paiement est lié à un Client et à l'Utilisateur qui a encaissé.

6. Sécurité et Gestion des Accès

La sécurité est assurée à plusieurs niveaux dans la conception :

  • Redirection automatique vers la page de login si $_SESSION['user_id'] n'existe pas dans les constructeurs des contrôleurs protégés.
  • Hachage des mots de passe avec l'algorithme BCRYPT via password_hash().
  • Protection contre les failles XSS en utilisant htmlspecialchars() dans toutes les vues PHP.
  • Protection contre les injections SQL via les requêtes préparées PDO.
  • Nettoyage des entrées avec trim() et validation des URLs avec filter_var().

Technologies Utilisées

Dans le cadre de la réalisation de 7X Pharma Nexus, un ensemble de technologies soigneusement sélectionnées a été utilisé pour garantir performance, sécurité et maintenabilité.

MySQL / MariaDB Gestion de la base de données relationnelle pharma_core.

1. PHP 8.2 — Back-end

PHP est le langage de programmation côté serveur au cœur de l'application. Utilisé dans sa version 8.2 avec une approche entièrement orientée objet, il permet de structurer proprement le code selon le patron MVC. Il gère l'authentification, les sessions, le routage et toute la logique métier de chaque module (POS, Inventaire, Crédits, Dashboard).

La classe de base Model.php fournit des méthodes génériques (getAll, getById, insert, update, delete) qui construisent dynamiquement des requêtes SQL préparées, respectant le principe DRY (Don't Repeat Yourself).

2. MySQL / MariaDB — Base de Données

MySQL est utilisé pour stocker toutes les informations persistantes du système : médicaments, lots, ventes, articles de vente, clients et mouvements de stock. La connexion est gérée via PDO dans Database.php, garantissant la sécurité des requêtes par l'utilisation systématique des paramètres liés.

Le projet exploite des requêtes SQL avancées : LEFT JOIN, GROUP BY, COALESCE, HAVING, et la clause de soft delete (deleted_at IS NULL).

Le back-end en PHP 8.2 constitue le cœur fonctionnel de l'application.

L'authentification est gérée par AuthController.php. Lors de la soumission du formulaire de connexion, le système appelle User->verifyUser(), qui utilise password_verify() pour comparer le mot de passe saisi avec le hash BCRYPT stocké. En cas de succès, les informations (id, nom, rôle) sont sauvegardées dans $_SESSION.

Gestion des Sessions :

Les sessions PHP sont utilisées pour :

  1. Maintenir l'utilisateur connecté via $_SESSION['user_id'].
  2. Gérer le panier d'achat temporaire du POS via $_SESSION['cart'], évitant ainsi de solliciter la base de données avant la validation définitive de la vente.
  3. Afficher des messages temporaires (Flash Messages) via les méthodes setFlash et getFlash du contrôleur principal.

Opérations CRUD — Module Inventaire :

  • Create : La méthode add() insère le médicament dans medicines, puis si un numéro de lot est fourni, insère également le lot dans batches.
  • Read : La méthode index() lit la liste des médicaments avec leurs quantités totales calculées via LEFT JOIN et GROUP BY.
  • Update : La méthode edit() met à jour les informations du médicament (nom, prix, DCI).
  • Delete : La méthode delete() utilise un Soft Delete en mettant à jour la colonne deleted_at avec la date actuelle, préservant l'intégrité historique.

Module POS — Logique de Checkout :

Le POSController est le module le plus complexe de l'application. Lors de la validation d'une vente, le système :

  1. Crée une entrée globale dans la table sales.
  2. Pour chaque article du panier, trouve le lot le plus ancien disponible (FIFO) via getOldestBatchId().
  3. Enregistre les détails de l'article dans sale_items avec un snapshot JSON du nom et du prix au moment de la vente.
  4. Déduit le stock via Medicine->deductStock(), en traversant automatiquement plusieurs lots si nécessaire.
  5. Si le paiement est "Crédit", ajoute la dette au compte client via Client->addDebt().

Le DashboardController agrège les données essentielles pour le gérant. Il appelle getTodayRevenue() de Sale.php pour les recettes du jour, et getLowStockAlerts() de Medicine.php en utilisant la clause SQL HAVING current_quantity <= 10.

Section VII

Tests et Validation

1. Introduction

Dans cette phase critique de développement, j'ai procédé à un audit complet du code source et de l'architecture du projet 7X Pharma Nexus. L'objectif était de valider la conformité des fonctionnalités implémentées par rapport aux besoins définis initialement. J'ai réalisé ces tests pour m'assurer que le système est robuste, que les règles de gestion métier sont respectées, et que la sécurité des données est garantie avant un déploiement en conditions réelles.

2. Méthodologie de Test

J'ai configuré un environnement de test local en utilisant la plateforme XAMPP (PHP 8.2 et MariaDB). Lors de mes revues de code, j'ai analysé directement l'architecture MVC personnalisée (Contrôleurs, Modèles, Vues) et exécuté des scénarios de test sur les différentes interfaces via Google Chrome. J'ai également examiné la structure de la base de données pharma_core.sql pour valider l'intégrité et la cohérence des données stockées. Mes scénarios de test couvraient l'ensemble du cycle de vie de l'application : de l'authentification jusqu'aux transactions les plus complexes du Point de Vente (POS).

3. Tests Fonctionnels

J'ai exécuté manuellement de nombreux scénarios pour valider les fonctionnalités clés de l'application :

  • Point de Vente (POS) : J'ai testé l'ajout de médicaments au panier, le calcul du total et le processus de paiement (checkout). Résultat attendu : création d'une facture, insertion dans les tables sales et sale_items, et déduction du stock. Résultat obtenu : le processus de checkout encapsulé dans une transaction PDO globale fonctionne correctement. Le snapshot JSON détaillé des articles est bien sauvegardé.
  • Gestion de l'inventaire : J'ai testé l'ajout, la modification et la suppression d'un médicament (via le InventoryController). Résultat attendu : mise à jour de la base de données sans supprimer définitivement l'enregistrement pour conserver l'historique. Résultat obtenu : le mécanisme de "soft delete" fonctionne parfaitement en mettant à jour le champ deleted_at. J'ai aussi validé l'unicité du code-barres grâce à la méthode barcodeExists().
  • Tableau de bord : J'ai vérifié le calcul des statistiques journalières (revenus, transactions) et les alertes de stock. Résultat attendu : affichage d'alertes pour les médicaments dont la quantité totale est inférieure ou égale à 10. Résultat obtenu : Les requêtes SQL utilisant la clause HAVING current_quantity <= 10 s'exécutent avec succès et les alertes sont bien remontées.

4. Tests des Formulaires

J'ai analysé et testé la soumission de plusieurs formulaires, notamment ceux d'authentification et d'ajout de médicaments dans l'inventaire.

  • Validation côté client : J'ai constaté que les champs obligatoires utilisent efficacement l'attribut HTML5 required, et que le fichier forms.css exploite les pseudo-classes :invalid et :valid couplées à la classe .was-validated pour afficher visuellement des bordures d'alerte rouges ou vertes.
  • Validation côté serveur : J'ai examiné la gestion PHP (par exemple dans AuthController::login et InventoryController::add). J'ai validé que le code effectue des vérifications basiques pour éviter les champs vides (trim()) et sécuriser les types numériques (floatval() et intval()). Cependant, j'ai constaté un manque de validation plus stricte (telle que l'utilisation d'expressions régulières) pour vérifier les formats précis des données saisies.

5. Tests de la Logique FIFO

J'ai testé rigoureusement l'algorithme de déduction des stocks en appliquant le principe FIFO (First In, First Out).

Scénario exécuté : J'ai déclenché la vente d'une quantité élevée d'un médicament, nécessitant de puiser dans plusieurs lots différents. Ensuite, j'ai analysé la méthode deductStock() du modèle Medicine.php.

Comportement attendu : L'algorithme doit déduire les unités du lot dont la date d'expiration est la plus proche, l'épuiser, puis déduire la quantité restante du lot suivant.

Comportement obtenu : J'ai confirmé que le système interroge les lots par ordre de date d'expiration (ORDER BY expiry_date ASC). La boucle déduit correctement la quantité du premier lot, puis passe au suivant si nécessaire, en mettant les lots épuisés à 0. L'algorithme FIFO est parfaitement implémenté et opérationnel.

6. Tests de Base de Données

J'ai analysé les requêtes SQL, la structure de la base et le comportement des modèles :

  • Intégrité relationnelle : J'ai vérifié les clés étrangères dans pharma_core.sql. J'ai validé que la suppression en cascade est active (ON DELETE CASCADE) pour la table sale_items liée à sales, ce qui maintient une base de données propre.
  • Cohérence des données : J'ai constaté un défaut d'architecture critique dans le schéma SQL concernant les crédits. La méthode makePayment() du modèle Client.php tente d'insérer des valeurs pour payment_method et note dans la table client_payments. Or, j'ai vérifié que ces deux colonnes sont totalement absentes de la définition de cette table. Cela provoque une défaillance silencieuse lors du règlement d'une dette client.

7. Tests de Sécurité

J'ai audité le code source sous l'angle critique de la sécurité informatique :

  • Authentification et mots de passe : J'ai validé que les mots de passe des utilisateurs sont correctement hachés avec l'algorithme BCRYPT via password_hash() et vérifiés via password_verify() dans la classe User.php. Les sessions sont initiées proprement (session_start()).
  • Injections SQL : J'ai confirmé que l'intégralité du système utilise des requêtes SQL préparées via l'extension PDO ($stmt = $this->db->prepare()), rendant l'application très résistante aux attaques par injection SQL.
  • Protection XSS (Cross-Site Scripting) : J'ai analysé les différentes vues (ex: dashboard.php, pos.php, inventory.php) et j'ai constaté que l'affichage des données provenant de la base de données ou des variables utilisateur est systématiquement sécurisé par la fonction PHP htmlspecialchars(), empêchant l'exécution de scripts malveillants.
  • Vulnérabilité CSRF (Cross-Site Request Forgery) : J'ai recherché des mécanismes de protection pour la soumission des formulaires. J'ai constaté qu'aucun jeton CSRF n'est généré ni vérifié côté serveur, laissant l'application vulnérable à la soumission malveillante de requêtes.
  • Contrôle d'accès : J'ai validé le fonctionnement de la méthode requireRoles() dans le contrôleur de base, qui sécurise efficacement les routes et redirige tout accès non autorisé en fonction des rôles (admin/pharmacist) stockés en session.

8. Tests Responsive

J'ai inspecté les principaux fichiers CSS (global.css, forms.css, dashboard.css, pos.css) et simulé diverses résolutions d'écran pour analyser l'adaptabilité de l'interface.

  • Desktop et Tablette : J'ai vérifié la présence et l'application de Media Queries ciblées (notamment @media (max-width: 1024px) et 768px). L'utilisation de la grille CSS permet à l'interface de se réorganiser de manière fluide, passant d'un affichage étendu à des colonnes réduites.
  • Mobile : J'ai confirmé que pour les écrans inférieurs à 480px, le menu de navigation latéral (Sidebar) se rétracte et s'affiche sous forme d'un menu d'overlay (géré en JavaScript via dashboard.js), offrant une navigation accessible et sans encombrement. L'application est globalement très réactive.

9. Résultats et Analyse

Mon audit technique révèle une application de gestion de pharmacie robuste et performante. L'architecture MVC implémentée nativement est claire, organisée et favorise grandement la maintenabilité du code.

Points forts : J'ai particulièrement apprécié l'implémentation réussie de la logique d'inventaire FIFO avec gestion des lots, l'utilisation judicieuse des transactions PDO globales pour éviter les erreurs partielles de facturation, et la protection généralisée contre les failles XSS. Le design UI/UX moderne, incluant une option de thème sombre, offre une expérience très agréable.

Limitations et fonctionnalités incomplètes : Le défaut structurel de la base de données (colonnes payment_method et note manquantes dans la table client_payments) est un dysfonctionnement critique qu'il faudra corriger. De plus, l'absence totale de jetons CSRF pour sécuriser la soumission des formulaires constitue un risque de sécurité notable. Enfin, des contrôles de validation côté serveur plus stricts, par exemple avec des expressions régulières, permettraient de prévenir les entrées mal formatées.
